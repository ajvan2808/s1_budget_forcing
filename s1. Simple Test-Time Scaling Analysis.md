## Phần 1: Tóm tắt Bài báo

### 1.1 Bài toán nghiên cứu & Động lực

- **Research problem:** Tìm cách tiếp cận **đơn giản nhất** (simplest approach) để đạt được cả hai mục tiêu: (a) test-time scaling — hiệu suất tăng khi dùng thêm compute tại inference, và (b) strong reasoning performance — hiệu suất suy luận mạnh trên các bài toán khó.
- **Motivation:** OpenAI o1 đã chứng minh tính khả thi của test-time scaling nhưng **không công khai phương pháp** (§1: *"o1 has demonstrated strong reasoning performance... OpenAI describes their approach as using large-scale RL... This has led to various attempts to replicate"*). Mặc dù có nhiều nỗ lực tái tạo (MCTS, multi-agent, RL), **chưa có công trình mã nguồn mở nào tái tạo thành công hành vi test-time scaling rõ ràng** (§1: *"none have openly replicated a clear test-time scaling behavior"*).
- **Tuyên bố chính:** Chỉ cần training trên **1,000 mẫu** với next-token prediction + kỹ thuật **budget forcing** đơn giản là đủ để xây dựng reasoning model mạnh có khả năng scaling.

### 1.2 Phương pháp chính

#### (a) Quy trình tạo dữ liệu s1K

Quy trình 3 giai đoạn theo 3 nguyên tắc — **Quality, Difficulty, Diversity** (§2):

**Thu thập ban đầu (59K samples):** Tổng hợp 59,029 câu hỏi từ 16 nguồn (NuminaMATH: 30,660, OlympicArena: 4,250, OmniMath: 4,238, AGIEval: 2,385, v.v.) + 2 dataset tự tạo (s1-prob: 182 câu, s1-teasers: 23 câu). Dùng Gemini Flash Thinking API để sinh reasoning trace cho mỗi câu hỏi.

1. **Lọc chất lượng (Quality):** Loại API errors → 54,116; loại formatting issues → 51,581; chọn 384 mẫu từ các dataset chất lượng cao.
2. **Lọc độ khó (Difficulty):** Đánh giá bằng 2 mô hình (Qwen2.5-7B-Instruct & Qwen2.5-32B-Instruct), loại bỏ các câu mà **bất kỳ mô hình nào** trả lời đúng → 24,496 mẫu.
3. **Lọc đa dạng (Diversity):** Phân loại theo MSC taxonomy (50 domains), lấy mẫu theo phương pháp power-law weighting ưu tiên reasoning trace dài hơn (Algorithm 1, §E.1) → **1,000 mẫu cuối cùng (s1K)**.

> [!NOTE]
> Đáng chú ý: Chỉ **53.6%** câu trả lời trong s1K là đúng (§2.2). Điều này cho thấy SFT học **reasoning patterns** chứ không phải correct answers (*"the SFT stage is about learning reasoning patterns rather than correct answers"* — §4.2).

#### (b) Kỹ thuật Budget Forcing

Budget Forcing là kỹ thuật can thiệp tại thời điểm decoding (decoding-time intervention) kiểm soát số lượng thinking tokens (§3.1):

- **Enforce maximum (giảm compute):** Chèn end-of-thinking token delimiter + "Final Answer:" để buộc mô hình dừng suy nghĩ sớm và đưa ra best guess hiện tại.
- **Enforce minimum (tăng compute):** Suppress end-of-thinking token delimiter + nối thêm chuỗi **"Wait"** vào reasoning trace để khuyến khích mô hình **self-reflect và double-check** câu trả lời. Ví dụ minh họa (Figure 3): mô hình ban đầu trả lời "raspBerry có 2 chữ r", sau khi bị buộc tiếp tục suy nghĩ bằng "Wait", mô hình tự sửa thành 3 chữ r.

## Phần 2: Phân tích Phương pháp Budget Forcing

### 2.1 Cơ chế hoạt động chi tiết

Budget Forcing hoạt động ở tầng **decoding** (không cần thay đổi mô hình hay retrain):

```
┌─────────────────────────────────────────────────┐
│                BUDGET FORCING                    │
│                                                  │
│  ┌── Enforce MAXIMUM ──┐  ┌── Enforce MINIMUM ──┐│
│  │ Khi tokens > budget: │  │ Khi model cố dừng: ││
│  │ 1. Append end-of-    │  │ 1. Suppress end-of- ││
│  │    thinking delimiter│  │    thinking token    ││
│  │ 2. Append "Final     │  │ 2. Append "Wait"    ││
│  │    Answer:"          │  │ 3. Model tiếp tục   ││
│  │ 3. Force early exit  │  │    reasoning, self-  ││
│  │                      │  │    check answer      ││
│  └──────────────────────┘  └─────────────────────┘│
└─────────────────────────────────────────────────┘
```

**Enforce minimum (tăng compute):** Khi mô hình sinh end-of-thinking token, token này bị **suppress** (không thêm vào output) và thay bằng **"Wait"**. Điều này buộc mô hình:
- Đọc lại reasoning trace hiện tại
- Self-reflect: kiểm tra lại logic
- Backtrack: phát hiện và sửa lỗi (như ví dụ "raspberry" trong Figure 3)

**Enforce maximum (giảm compute):** Khi đạt ngưỡng token tối đa, chèn end-of-thinking delimiter + "Final Answer:" để model đưa ra best guess ngay lập tức, tránh lãng phí compute.

### 2.3 Phân tích 3 Metrics đánh giá

**① Control (Eq. 1, §3.2):** Đo mức độ **kiểm soát** lượng compute thực tế so với target. 100% = perfect control.
- BF đạt **100%** vì trực tiếp can thiệp vào quá trình generation (suppress/append tokens), không phụ thuộc mô hình tự kiểm soát.
- TCC chỉ 40%: mô hình **không biết đếm tokens** chính xác — kể cả khi được train để đếm (§5.2: *"our model cannot reliably count tokens - even when trained to do so"*).
- SCC chỉ 60%: mô hình "hack" constraint — ít bước nhưng mỗi bước dài hơn, hoặc ngược lại (§5.2: *"the model learns to hack its way around the compute constraint"*).

**② Scaling (Eq. 2, §3.2):** Trung bình slope của đường cong accuracy-vs-compute. Phải **dương** và càng lớn càng tốt.
- BF: **+15** — clear positive scaling trend
- TCC: **-24** — inverse scaling (tệ hơn khi dùng nhiều compute!)
- RS: **-35** — **inverse scaling nghiêm trọng nhất**. Tác giả giải thích (§5.2, Figure 6): các generation ngắn thường đúng vì mô hình "on the right track from the start"; generation dài thường sai vì backtrack nhiều. Rejection sampling cho generation dài → chọn nhầm output sai.
- CCC: +25 (cao nhất) nhưng **chỉ chạy 2 runs** nên kém robust.

**③ Performance (Eq. 3, §3.2):** Accuracy tối đa đạt được trên benchmark.
- BF: **56.7%** — cao nhất. Budget Forcing không chỉ scale tốt mà còn đạt performance tuyệt đối cao nhất.

### 2.4 Tại sao Budget Forcing vượt trội?

1. **Kiểm soát trực tiếp tại decoding-level:** Không yêu cầu mô hình tự ước lượng compute → không bị "hack" hay đếm sai.
2. **Cơ chế self-reflection tự nhiên:** "Wait" trigger mô hình re-read reasoning trace → phát hiện lỗi logic tương tự human double-checking.
3. **Flexibility:** Kết hợp cả max và min enforcement trong cùng framework, hoạt động với bất kỳ reasoning model nào mà không cần retrain.
4. **Đơn giản triển khai:** Chỉ cần thay đổi decoding loop, không cần reward model (REBASE) hay multiple sampling (majority voting).

## Phần 3: Đánh giá Điểm mạnh & Điểm yếu

### 3.1 Điểm mạnh

1. **Tính đơn giản vượt trội (Simplicity):** Toàn bộ pipeline — từ data curation đến training đến inference — cực kỳ đơn giản. SFT trên 1K mẫu + budget forcing tại decoding. Không cần RL, reward model, hay MCTS.

2. **Sample-efficiency đáng kinh ngạc:** 1,000 mẫu training đạt hiệu suất cạnh tranh với r1-distill dùng 800K mẫu (gấp 800×). Finetuning chỉ **26 phút trên 16 H100** = 7 GPU hours (§4.1).

3. **Framework đánh giá test-time scaling:** Bài báo đề xuất 3 metrics định lượng (Control, Scaling, Performance) — đây là đóng góp quan trọng cho cộng đồng nghiên cứu, tạo chuẩn so sánh cho các phương pháp tương lai (§3.2).

4. **Hoàn toàn mã nguồn mở:** Model weights, data (s1K), code — tất cả đều public. Trong bối cảnh o1 đóng và R1 chỉ mở trọng số, đây là đóng góp đáng kể cho tính minh bạch (Ethical considerations: *"Our work aims to push the frontier of reasoning in a fully open manner"*).

5. **Ablation studies kỹ lưỡng:** Table 2 (data ablations), Table 3 (method ablations), Table 7 (sequence length ablation) cung cấp bằng chứng thuyết phục cho từng thiết kế. Bootstrap confidence intervals trong Table 2 tăng tính nghiêm ngặt thống kê.

6. **Insight quan trọng về SFT:** Phát hiện rằng **~50% câu trả lời sai trong training data vẫn cho kết quả tốt** — gợi ý SFT học reasoning process chứ không chỉ correct answers.

### 3.2 Điểm yếu & Hạn chế

1. **Chỉ thử nghiệm trên Qwen family:** Toàn bộ thí nghiệm chỉ dùng Qwen2.5-32B-Instruct (§Limitations: *"While we only experiment with models from the Qwen family"*). Tác giả dẫn follow-up works (Yong et al., 2025) nhưng **bản thân paper không chứng minh generalization**.

2. **Budget Forcing flatlines & context window:** Kỹ thuật bị giới hạn: (a) scaling eventually flattens out (Figure 4 — sau 6x "Wait"), (b) bị ràng buộc bởi context window (§6.2: 12/30 AIME24 questions vượt context length). Tác giả thừa nhận *"it has two key limitations when scaling further"*.

3. **Repetitive loops:** Suppress end-of-thinking quá nhiều lần → mô hình rơi vào repetitive loops thay vì continued reasoning (§4.2: *"Suppressing the end-of-thinking token delimiter too often can lead the model into repetitive loops"*).

4. **Phụ thuộc vào distillation từ model mạnh:** s1K dùng Gemini traces, s1K-1.1 dùng DeepSeek r1 traces (§Limitations — Distillation: *"This could be a limitation as it assumes that a powerful model is accessible"*). Dù tác giả gợi ý dùng human experts thay thế, chưa được kiểm chứng.

5. **Phạm vi task hạn hẹp:** Chỉ đánh giá trên math/science reasoning (AIME, MATH, GPQA). Tính hiệu quả trên abstract tasks (creative writing, coding) **chưa được kiểm chứng** (§Limitations: *"One major limitation... is their applicability to abstract tasks, such as creative writing"*).

6. **Benchmark size nhỏ:** AIME24 chỉ có **30 câu hỏi** — variance cao, kết quả có thể không stable. Sự khác biệt 1-2 câu = 3-7% accuracy.

7. **Chưa có human evaluation:** Tất cả đánh giá đều automatic. Chất lượng reasoning trace (coherence, depth) chưa được đánh giá bởi human experts.

### 3.3 Tính khả thi tái tạo (Reproducibility)

- **Cao:** Code, data, model weights đều open-source. Training hyperparameters rõ ràng (§F). Pipeline dùng PyTorch FSDP chuẩn.
- **Hạn chế:** Cần 16 H100 GPUs cho full reproduction. Tuy nhiên, có thể thử trên model nhỏ hơn (Qwen2.5-7B) với ít GPU hơn.
- **Evaluation:** Dùng lm-evaluation-harness framework (§4.1), greedy decoding (temperature=0) — dễ tái tạo deterministic.

### 3.4 Tính tổng quát hóa (Generalizability)

- **Model family:** Tác giả chỉ thử trên Qwen. Follow-up works (cited trong Limitations) cho thấy generalizes across scales nhưng **chưa chứng minh trên Llama, Mistral, hay các family khác**.
- **Task type:** Chỉ math/science. Budget Forcing có lý thuyết hoạt động được cho abstract tasks (vì không dùng majority voting) nhưng chưa kiểm chứng.
- **Language:** Chưa thử với non-English reasoning.

### 3.5 Giả định ẩn (Implicit Assumptions)

1. **"Thinking longer = thinking better":** Giả định rằng thêm tokens = thêm reasoning depth. Nhưng repetitive loops (§4.2) cho thấy điều này không luôn đúng.
2. **"Wait" trigger self-reflection:** Giả định rằng token "Wait" đủ để trigger re-evaluation. Chưa phân tích tại sao "Wait" cụ thể hoạt động tốt hơn các trigger khác.
3. **Reasoning traces from strong models contain transferable patterns:** Giả định rằng reasoning patterns từ Gemini/R1 transferable sang Qwen qua SFT.
4. **Difficulty ≈ trace length:** Dùng trace length làm proxy cho difficulty. Có thể có mismatch giữa "dài" và "khó".

## Phần 4: Khoảng trống Nghiên cứu (Research Gaps)

### Gap 1: Budget Forcing trên các model families khác
- **Mô tả:** Paper chỉ thử trên Qwen family. Liệu Budget Forcing có hoạt động tương tự trên Llama 3, Mistral, Phi, Gemma?
- **Mức độ ảnh hưởng:** ⭐⭐⭐⭐⭐ (Rất cao — cần thiết để validate tính tổng quát)
- **Khả thi với tài nguyên hạn chế:** ⭐⭐⭐⭐⭐ (Rất khả thi — chỉ cần inference trên các model nhỏ 7B-14B, không cần retrain)

### Gap 2: Tối ưu chiến lược append cho Budget Forcing
- **Mô tả:** Paper chỉ dùng **"Wait"** cố định. Có trigger khác tốt hơn không? ("Let me reconsider", "Hold on", "Double-check"?) Tần suất append tối ưu? Có nên dùng adaptive strategy thay vì fixed? (§6.2 gợi ý: *"combining it with frequency penalties or higher temperature"*)
- **Mức độ ảnh hưởng:** ⭐⭐⭐⭐ (Cao — trực tiếp cải thiện Budget Forcing)
- **Khả thi với tài nguyên hạn chế:** ⭐⭐⭐⭐⭐ (Rất khả thi — chỉ cần thay đổi prompt/decoding, không cần train)

### Gap 3: Budget Forcing trên non-math tasks
- **Mô tả:** Paper thừa nhận hạn chế trên abstract tasks (§Limitations: *"applicability to abstract tasks, such as creative writing"*). Liệu BF có hiệu quả cho code generation, logical reasoning, commonsense reasoning?
- **Mức độ ảnh hưởng:** ⭐⭐⭐⭐ (Cao — mở rộng applicability)
- **Khả thi với tài nguyên hạn chế:** ⭐⭐⭐⭐ (Khả thi — benchmark cho code/logic reasoning đã có sẵn)

### Gap 4: Kết hợp Budget Forcing với RL-trained models
- **Mô tả:** §6.2 gợi ý: *"whether applying budget forcing to a reasoning model trained with reinforcement learning yields better extrapolation"*. s1 dùng SFT, nhưng RL models (R1, o1) có reasoning pattern khác — liệu BF có synergy?
- **Mức độ ảnh hưởng:** ⭐⭐⭐⭐⭐ (Rất cao — potential breakthrough)
- **Khả thi với tài nguyên hạn chế:** ⭐⭐⭐⭐ (Khả thi — dùng r1-distill models đã open-weight, chỉ cần thay decoding)

### Gap 5: Giảm repetitive loops trong extended thinking
- **Mô tả:** Hạn chế chính: append "Wait" nhiều lần → repetitive loops (§4.2). Cần cơ chế **phát hiện và thoát loops** hoặc **diversity enforcement** (frequency penalty, temperature scaling). Paper gợi ý nhưng chưa thực hiện (§6.2: *"combining it with frequency penalties or higher temperature to avoid repetitive loops"*).
- **Mức độ ảnh hưởng:** ⭐⭐⭐⭐ (Cao — giải quyết bottleneck chính)
- **Khả thi với tài nguyên hạn chế:** ⭐⭐⭐⭐⭐ (Rất khả thi — chỉ cần thay đổi decoding strategy)

### Gap 6: Phân tích cơ chế tại sao Budget Forcing hoạt động
- **Mô tả:** Paper cho thấy BF works nhưng **chưa giải thích WHY** ở mức cơ chế. "Wait" kích hoạt attention pattern nào? Mô hình thực sự "suy nghĩ lại" hay chỉ sinh thêm text?
- **Mức độ ảnh hưởng:** ⭐⭐⭐ (Trung bình-cao — interpretability)
- **Khả thi với tài nguyên hạn chế:** ⭐⭐⭐ (Trung bình — cần phân tích attention weights, probing)

### Gap 7: Adaptive Budget Allocation
- **Mô tả:** Paper dùng fixed budget cho tất cả câu hỏi. Câu hỏi dễ cần ít compute, câu khó cần nhiều. Một chiến lược **adaptive** phân bổ budget dựa trên confidence/uncertainty có thể hiệu quả hơn.
- **Mức độ ảnh hưởng:** ⭐⭐⭐⭐ (Cao — tiết kiệm compute + tăng accuracy)
- **Khả thi với tài nguyên hạn chế:** ⭐⭐⭐⭐ (Khả thi — cần heuristic đơn giản cho confidence estimation)

### 5.2 Câu hỏi nghiên cứu

**Câu hỏi chính (RQ):**
> Budget Forcing có tổng quát hóa hiệu quả trên các model families (ngoài Qwen) và các chiến lược append trigger khác nhau không? Nếu có, cấu hình nào tối ưu cho từng loại mô hình?

**Câu hỏi phụ:**
- **RQ1:** Hiệu quả của Budget Forcing thay đổi như thế nào khi áp dụng trên các model families khác nhau (Qwen, Llama, Gemma, Phi) ở cùng kích thước tham số?
- **RQ2:** Các trigger phrases thay thế ("Wait", "Let me reconsider", "Hmm, hold on", custom) ảnh hưởng như thế nào đến chất lượng self-reflection và test-time scaling slope?
- **RQ3:** Kết hợp Budget Forcing với các kỹ thuật chống repetitive loops (frequency penalty, temperature annealing) có cải thiện extrapolation performance không?

### 5.3 Giả thuyết nghiên cứu

- **H1:** Budget Forcing duy trì positive scaling trend (Scaling > 0) trên **ít nhất 3/4 model families** được thử nghiệm, với mức Performance trong khoảng ±10% so với kết quả trên Qwen.
- **H2:** Tồn tại ít nhất một trigger phrase thay thế đạt Scaling cao hơn "Wait" trên ít nhất 1 benchmark.
- **H3:** Kết hợp Budget Forcing + frequency penalty giảm tỷ lệ repetitive loops ≥30% so với BF đơn thuần và duy trì hoặc cải thiện Performance.

#### Chi tiết từng phase:

**Phase 1 — Baseline Reproduction (Tuần 1-2):**
- Load dataset s1K từ GitHub repository
- SFT Qwen2.5-7B-Instruct trên s1K bằng QLoRA (4-bit quantization + LoRA rank 64) để fit vào Colab
- Reproduce kết quả Budget Forcing trên MATH500 (500 câu, đủ cho statistical significance)
- Xác nhận pipeline hoạt động đúng

**Phase 2 — Cross-Model Generalization (Tuần 3-5):**

| Model | Kích thước | Cách tiếp cận | Dự kiến VRAM |
|-------|-----------|---------------|-------------|
| Qwen2.5-7B-Instruct | 7B | SFT (QLoRA) trên s1K → BF | ~16GB (4-bit) |
| DeepSeek-R1-Distill-Qwen-14B | 14B | **Chỉ inference** + BF (đã trained reasoning) | ~12GB (4-bit) |
| Llama-3.1-8B-Instruct | 8B | SFT (QLoRA) trên s1K → BF | ~16GB (4-bit) |
| Gemma-2-9B-IT | 9B | SFT (QLoRA) trên s1K → BF | ~16GB (4-bit) |

> [!IMPORTANT]
> DeepSeek-R1-Distill-Qwen-14B là candidate đặc biệt quan trọng: đã được train cho reasoning qua RL distillation → kiểm tra Gap 4 (BF + RL-trained model).

**Phase 3 — Trigger Optimization (Tuần 6-7):**
- Chọn **2 best models** từ Phase 2
- Test 5-7 trigger phrases: "Wait", "Hmm, let me reconsider", "Hold on, I need to verify this", "Let me double-check my reasoning", "Actually, wait..."
- Mỗi trigger: chạy BF với 1x, 2x, 4x append
- Đo Control, Scaling, Performance trên MATH500
- Phân tích: trigger nào tạo "productive self-reflection" vs "repetitive continuation"?

**Phase 4 — Anti-Repetition Enhancement (Tuần 8-9):**
- Implement repetition detection: n-gram overlap > threshold → trigger alternative action
- Test kết hợp: BF + frequency_penalty, BF + temperature annealing, BF + repetition-aware early stop
- So sánh với BF vanilla

### 5.6 Baseline cần so sánh
